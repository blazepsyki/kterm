# kterm 터미널 에뮬레이터 전체 기능 고도화 및 버그 해결 명세서

본 문서는 `kterm` (Iced 기반 네이티브 터미널 에뮬레이터) 개발 과정(v1~v13)에서 겪었던 주요 UI/UX 개선 및 백엔드 파이프라인의 고질적 터미널 버그 수정 내역을 하나로 집대성한 완전판 요약 보고서입니다.

---

## 1. 터미널 UI/UX 고도화 및 네이티브 기능 탑재

### 1-1. 모던 헤더 및 창 타이틀 바 통합
단순한 형태의 터미널을 넘어 윈도우 네이티브 프레임을 숨기고(`decorations: false`), 크롬 브라우저와 같은 보더리스(Borderless) 디자인의 통합 헤더를 구현했습니다.
- 마우스 드래그를 통해 창을 자유롭게 이동시키고 최소화/최대화/닫기 윈도우 컨트롤을 탑재했습니다.
- 배경에는 반투명 오버레이를 팝업할 수 있는 스택(`stack!`) 기반의 모던 아키텍처를 세팅했습니다.

### 1-2. 텍스트 선택(Selection) 및 마우스 인터랙션 완성
단순히 보기만 하는 터미널을 실질적인 터미널 환경으로 향상시키는 핵심 기능이었습니다.
- **선택 로직 및 드래그 버그 수정**: 마우스 클릭 좌표를 터미널 논리 행/열로 매핑했습니다. 초기에 "마우스 버튼을 떼어도 무한히 드래그 판정이 유지되는" 버그가 발생했으나, 위젯(`LocalState`)단에서 `is_dragging` 상태값을 명시적으로 추적하여 즉시 중단되도록 완벽히 교정했습니다.
- **클립보드 및 단축키 연동**: 시스템 클립보드와 통신하여 텍스트 데이터의 송수신을 구축했습니다.
- **ESC 논리적 우선순위 정립**: ESC 키를 눌렀을 때, 텍스트가 선택된 상태라면 **'터미널의 선택 해제'**만 이루어지고 쉘로 시그널이 넘어가지 않도록 방화벽 기능 우선순위를 정립했습니다. `Ctrl+C` 역시 선택 영역이 있을 때는 복사(`COPY`)가, 없을 때는 프로세스 종료(`SIGINT`)가 전송되도록 정교화했습니다.

## 2. 텍스트 입력 및 화면 렌더링 버그 수정의 여정

### 2-1. 한글 이중 점프 (`Double-Skip`) 버그
- **증상**: 한글 및 와이드 문자를 입력 후 좌우 방향키로 이동하거나 백스페이스로 지울 때, 글자가 2개씩 지워지거나 커서가 2칸씩 널뛰기하는 현상.
- **해결**: SSH 접속 시 백엔드 쉘(ConPTY)이 이미 자체적으로 2회 지우기를 보정하고 있다는 사실을 발견하고, 터미널 에뮬레이터 코드 내의 와이드 캐릭터용 `skip` 로직을 과감히 제거하여 제어권을 일원화했습니다.

### 2-2. 빈 공간 문자 할당 및 와이드 캐릭터 클린업 (`Wide-Aware Cleanup`)
- **증상**: 한글 등 2칸(Wide-character)을 차지하는 문자의 일부분만 덮어쓰거나 지울 경우(예: 화살표 이동 후 글자 타이핑), 반쪽짜리 빈 유령 널(`\0`) 문자가 남아 렌더링이 깨지는 현상.
- **해결**: 글자를 출력(`print`), 덮어쓰기, 삭제(`DCH/ICH/Clear`)할 때 원본 셀이 와이드 캐릭터 묶음이었다면 인접한 앞/뒤 `\0` 셀을 능동적으로 추적해 공백(`' '`)으로 강제 초기화하는 방어 로직을 전방위 도입했습니다.

### 2-3. 리사이즈 밀림 및 백스페이스 역래핑(Reverse-wrap) 파괴 버그
- **증상 1 (리사이즈)**: 터미널 폭을 좁힐 때 한글이 경계에 닿으면 허리가 분리된 채 강제로 개행되어 렌더링이 산산조각 났습니다.
- **증상 2 (백스페이스 역행 불가)**: 백스페이스로 긴 명령을 지워 올라가다 멀티행을 역추적하지 못하고 첫 번째 줄에서 막혀버렸습니다. (이는 잔상 문제의 핵심 발단 중 하나였습니다)
- **해결**: 글자가 찍힐 때 래핑 경계를 잘못 초기화하던 `line.is_wrapped = false` 하드코딩 오류를 제거하여 래핑 체인을 정상 복원했습니다. 또한, 우측 공간이 1칸만 남으면 한글 전체를 무조건 다음 행의 첫 칸으로 넘기는 무결성 보장 리플로우(Reflow) 로직을 개발했습니다. 더불어 백스페이스 시 `cursor_x = cols - 1`, `cursor_y -= 1`을 통한 역방향 래핑 롤백 로직을 완벽히 동기화했습니다.

### 2-4. 렌더링 캐시(Canvas) 시각적 멈춤 현상
- **증상**: 쉘이 명령을 통해 화면 부분을 지워도 화면에 여전히 글자가 남아있다가 키보드를 누르면 늦게 사라지는 지연 현상.
- **해결**: 내부 메모리(`Grid`)는 갱신되지만 UI를 그리는 Iced 엔진의 캐시를 무효화하는 명령(`cache.clear()`)이 없던 치명적 결함을 찾아, SSH 이벤트 파이프라인 수신부에 즉각 캐시 비우기 호출을 삽입했습니다.

## 3. 최종 보스: ConPTY 복합 다중 행 잔상 (Ghosting) 완벽 해소

터미널 하단에 끊임없이 고아 찌꺼기 문자열을 영구적으로 남기던 최고 난이도의 잔상 버그입니다. 원인은 Windows SSH 기반인 ConPTY의 델타 변경 최적화 렌더링 엔진 결함(와이드 캐릭터 길이 산정 착오 및 \r 커서 롤백 누락)이었습니다. OS 레벨 서버 버그를 클라이언트(kterm 터미널) 수준에서 스마트하게 뒷수습하는 3단계 전술을 완성했습니다.

### 💡 능동형 지우기 우회 및 방어막 (Heuristics)
1. **래핑 고아행(Orphan Wrapped Lines) 추적 소거**
   쉘이 명령을 대체하기 위해 위 화면 좌측으로 강제 커서 점프(`CUP`)를 수행했을 때 터미널이 래핑 속성(`is_wrapped=true`) 체인을 즉시 역추적하여, 이전 쉘 명령 찌꺼기로 추정되는 라인들을 화면 렌더링 직전에 자체 클리어(`Clear Entire Line`)합니다.
2. **무의미한 델타 패턴(`X n + C n`) 필터링 소거**
   개행 프롬프트(`>>`) 같은 비 래핑 상태에서, 커서 이동도 없이 허공 뒤쪽을 향해 "지운 후 이동해라"라며 던지는 비정상 패턴을 감지하여 전체 줄 클리어로 격상시켰습니다.
3. **새 텍스트 인쇄 상태 기반 정밀 폭격 (HashSet Tracking)**
   쉘이 점프 후 `CSI X`(Erase) 파편 소거 명령을 내릴 때, 터미널이 **"우리가 이 행에 방금 새 글자나 프롬프트를 그린 적이 있는가?" (`history_printed_lines`)**를 메모리 해시맵으로 면밀히 추적합니다.
   - 글자를 쓰지 않은 빈 행에 지우기 명령이 오면 100% 잔상이므로 깡통 행을 통째로 파괴(`EL 2`).
   - 새 글자(새 프롬프트)를 쓰고 난 행에 도착한 지우기 명령이라면 유효 텍스트를 살리고 뒤의 한글 오버행 부분만 정밀 절단(`EL 0`).

### 🎉 결론
오로지 화면상의 문자와 이스케이프 시퀀스의 기형적 논리 패턴만을 분석하여 터미널과 서버 간 동기화를 무너뜨렸던 가장 악랄했던 십여 개의 버그 사슬을 완벽하게 치료해냈습니다.

## 4. RDP/XRDP NumLock 불일치 이슈와 현재 타협안

RDP 세션, 특히 XRDP + LXQt 조합에서 로그인 화면 이후 GUI가 로드된 뒤 첫 NumPad 입력이 숫자가 아닌 방향키/내비게이션 키로 해석되는 문제가 확인되었습니다.

### 4-1. 증상
- 로컬 Windows 측 NumLock은 ON 상태입니다.
- XRDP 로그인 화면에서는 sync가 정상 동작하는 것처럼 보이지만, LXQt 데스크톱이 올라온 직후 첫 NumPad 입력은 원격에서 `Home/End/Arrow/Insert/Delete` 계열로 처리됩니다.
- Lock 키를 한 번 누르거나 포커스를 전환해 다시 sync를 보내면 이후에는 정상 동작합니다.

### 4-2. 분석
- `TS_SYNC_EVENT`, `DeactivateAll`, `SetKeyboardIndicators`를 활용한 프로토콜 기반 복구 가능성을 우선 검토했습니다.
- 이를 위해 RDP PDU 추적 로그를 추가해 연결/런타임 구간을 확인했습니다.
- 결과적으로 테스트한 XRDP(LXQt) 서버는 로그인 화면 → 데스크톱 전환 시 `DeactivateAll`, `SetKeyboardIndicators`, 추가 `SaveSessionInfo`를 보내지 않았습니다.
- 즉, 원격 X11 세션이 NumLock 상태를 내부적으로 재설정해도 클라이언트는 이를 RDP 이벤트로 감지할 수 없었습니다.

### 4-3. 적용한 방식
- 모든 키다운 전에 sync를 보내는 방식은 동작하지만 범위가 과도했습니다.
- 최종적으로 NumLock 해석에 직접 영향을 받는 NumPad/Navigation 충돌 스캔코드(`0x47..0x53`)에 한해, key-down 직전에 `TS_SYNC_EVENT`를 먼저 전송하도록 정리했습니다.
- 결과적으로 문제의 직접 원인인 `NumPad 0-9`, `Decimal`, `Home/End/PageUp/PageDown/Insert/Delete/Arrow` 충돌 영역만 보정합니다.

### 4-4. 한계
- XRDP가 별도 전환 PDU를 보내지 않는 이상, “LXQt가 정확히 올라온 순간”을 프로토콜 차원에서 식별할 수 없습니다.
- 따라서 현재 방식은 완전한 자동 복구가 아니라, 문제가 발생하는 입력 직전에만 상태를 맞추는 실용적 절충안입니다.
- 일반 문자 키나 다른 lock-state 변화까지 모두 커버하는 보편 해법은 아닙니다.

## 5. RDP NSCodec 대응 전략 변경 (공식 지원 대기)

IronRDP가 아직 NSCodec을 공식 지원하지 않는 상태에서, kterm 독자 fallback 디코더를 유지하는 대신 안정성 우선 정책으로 방향을 변경했습니다.

### 5-1. 현재 방침
- NSCodec fallback 구현은 제거했습니다.
- NSCodec 협상도 비활성화하여, 공식적으로 처리 가능한 코덱 경로만 사용합니다.

### 5-2. 추후 계획
- IronRDP가 NSCodec을 공식 지원하면 해당 경로를 우선 반영합니다.
- 필요 시 그 이후에만 kterm 보조 fallback 도입 여부를 재평가합니다.

### 5-3. 기대 효과
- 비공식 우회 경로 유지 비용과 상호운용성 리스크를 줄입니다.
- 업스트림(IronRDP)과의 동작 일치성을 높여 유지보수 부담을 낮춥니다.

## 6. RDP 초기 화면 검정(첫 접속) 이슈 재현/원인/해결 (2026-03-28)

### 6-1. 재현 증상
- 첫 RDP 접속 직후 화면이 검게 보이거나 일부만 표시됩니다.
- VNC 탭을 한번 열었다가 RDP 탭으로 돌아오면 정상 표시되는 경우가 있었습니다.

### 6-2. 로그 기반 사실 확인
- 렌더러는 초기부터 정상 크기 viewport/scissor로 draw를 수행했습니다.
- 문제 세션에서 첫 유의미한 프레임 배치는 아래와 같이 `Full` 없이 `Rect`만 대량 수신되었습니다.
   - `[RDP-UI] session_id=0 frames=292 full=0 rect=292`
- 기존 경로에서는 이 초기 대량 rect 배치가 부분 업로드로 귀결될 수 있어, 부트스트랩 시점에 완전한 기준 프레임이 GPU에 올라가지 않는 케이스가 있었습니다.

### 6-3. 최종 수정
- `ConnectionEvent::Frames` 처리에서 부트스트랩 가드 추가:
   - 첫 유효 배치(`frame_seq_before == 0`)가 `rect-only`인 경우 `full_upload = true` 강제
   - `full=0 && rect>256`인 대배치도 동일하게 `full_upload = true` 강제
   - 강제 시 `dirty_rects.clear()`로 부분 업로드 대신 전체 버퍼 업로드를 유도
- 이 변경으로 첫 접속에서도 누적 CPU 버퍼가 즉시 전체 텍스처로 업로드되어 정상 화면이 표시됨을 확인했습니다.

### 6-4. 디버그 정책
- 디버그 로그는 기본 비활성화로 전환했습니다.
- 필요 시 환경변수 `KTERM_RDP_TRACE=1`로만 아래 로그를 활성화합니다.
   - `[RDP-UI] ...`
   - `[RDP-RENDER][prepare#...]`, `[RDP-RENDER][render#...]`

### 6-5. 검증 로그 스냅샷 (해결 확인)
- 초기 배치에서 rect-only 대량 업데이트가 들어온 뒤 강제 full upload가 즉시 동작함:
   - `[RDP-UI] session_id=0 frames=292 full=0 rect=292`
   - `[RDP-UI] force_full_upload session_id=0 reason=bootstrap_or_large_rect_batch rects=292`
   - `[RDP-RENDER][prepare#16] ... mode=full full_upload=true ...`
- 이후 prepare/render는 `mode=skip`으로 안정 상태를 유지하며 정상 화면이 유지됨을 확인했습니다.
